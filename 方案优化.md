# SecureClaw 方案优化（执行版）

本文档将 `idea_1.md` 与 `idea_2.md` 的新增内容，落地为当前仓库可执行、可验证、可写入论文主文/附录的统一方案。

## 0. 主命题与边界

主命题（当前可发表口径）：

- 主线：`NBE + SM + PEI + SCS + DAS`
- 可选增强：`SAP`（在双策略服务不合谋且固定形状泄露口径成立时）

其中：

- `NBE`：副作用提交不可绕过（Executor 双证明 + 绑定校验）
- `SM`：默认明文近视（handle-first + 显式 declassification）
- `PEI`：补丁携带外发完整性（sanitize patch 在 COMMIT 强制执行）
- `SCS`：skill 供应链与运行时受 capsule 合同约束
- `DAS`：多主体委托与联邦入口绑定安全（`hctx` 入哈希）
- `SAP`：单策略服务视角可模拟 transcript 隐私（可选模式）

## 1. 统一机制骨架

### 1.1 执行线与绑定

- 统一入口：`act(intent_id, inputs, constraints, caller)`
- 事务模型：`PREVIEW -> COMMIT`
- 请求绑定：`ReqHash(rho)`，其中 `rho=(intent_id, caller, session, inputs_eff, hctx)`
- `hctx` 至少包含：`external_principal`、`delegation_jti`

### 1.2 执行器验收谓词（Accept）

COMMIT 必须同时满足：

- 双策略服务 proof（`P0/P1`）MAC 验证通过
- `(action_id, program_id, request_sha256)` 一致
- freshness + anti-replay 通过
- `allow_pre/need_confirm` 位约束通过
- 若存在 patch 约束，则 sanitize patch 被强制执行

### 1.3 明文控制

- 高敏输入默认 handle 化，不向 runtime 直接返回明文
- 明文释放仅走显式 declassification 流程
- 去密级事件审计记录与预算扣减可追踪

## 2. 四个方向的落地方案（A/B/C/D）

## A. SAP + constant-shape policy outsourcing（可选主张）

### A1. 机制

- PIR：bundle + shift index + fixed key count
- MPC：统一 `program_id` 与固定 batch shape
- confirm path：DFA 固定步数（建议 `PAD_DFA_STEPS=1`）
- 时间通道：tick/bucket 级泄露，避免细粒度时间直出

### A2. 形式化口径

- 目标：单策略服务视图可由 `L_policy=(L_PIR,L_MPC,L_CONFIRM,L_TIME)` 模拟
- 条件：PIR 单方安全 + MPC 单方隐私 + 至少一台策略服务不合谋 + 固定形状约束满足

### A3. 实验与消融

- 指标：MI、单服务器 transcript 分类准确率（对比 chance）
- 消融：
  - `USE_POLICY_BUNDLE` on/off
  - `UNIFIED_POLICY` on/off
  - mixer on/off
  - `PAD_DFA_STEPS` 0/1
- 结论要求：full shaping 时可分离信号显著坍塌

## B. SM + Declassification + PEI（非 SAP 主线）

### B1. 机制

- Handle record 绑定：`session/caller/ttl/sensitivity/allowed_sinks`
- 明文释放接口：`Declassify(handle, purpose, confirm, policy_context)`
- Executor 强制补丁：`sanitize(payload, patch)` 后才允许 effect

### B2. 形式化口径

- `Bad_SM`：无合法去密级事件却实现明文区分或泄露
- `Bad_PEI`：Accept 成功但提交 payload 不满足 patch 约束
- 组合：`SM + PEI` 共同保证“明文默认不出 + 必要时可控释放 + 提交不可绕过补丁”

### B3. 实验

- 通道口径：`C1..C7` 泄露评估
- 指标：attack leak rate、benign allow、declassify 触发率、budget 触顶行为
- 关键展示：
  - 无 declassify 时高敏 token 不出现在 C1..C5
  - patch 违规提交被 Executor 拒绝

## C. Skill ingress + SCS（供应链与运行隔离）

### C1. 机制

- 安装流程 effect 化：
  - `ImportSkill -> DescribeSkill -> CheckSkillInstallPolicy -> CommitSkillInstall`
- IOC/install marker 纳入策略特征
- 启用后 caller 收敛到 `skill:<digest>`（最小权限投影）

### C2. 合同化系统假设

- `MC_CONTRACT_SPEC` 作为 machine-checkable 合同
- mandatory assertions 失败时：
  - 禁止声称 `SCS`
  - 进入降级语义（保留 NBE/SM/PEI/DAS/SAP 可选）

### C3. 实验

- 攻击集：host secret read、任意 exec、公网外发、loopback exfil
- 对照：无 capsule / 弱 capsule / 合同通过 capsule
- 验收：合同通过时 bypass 成功率近零；失败时系统显式降级

## D. Delegation + Dual-subject + Federated ingress binding

### D1. 机制

- 委托 token：`delta=(iss,sub,scope,session,exp,jti)`
- Gateway 校验：签名、session、scope、revocation、时效
- 权限交集：`effective_caps = caps(caller) ∩ caps(external_principal)`
- `hctx` 绑定进 `ReqHash(rho)` 防跨主体/跨会话复用

### D2. 形式化口径

- `Bad_DAS`：不同 `(caller, session, hctx)` 下复用旧证据仍被接受
- 归约：签名不可伪造 + 哈希绑定 + 规范序列化一致性 + replay/freshness 假设

### D3. 实验

- case 集：
  - missing delegation
  - revoked token replay
  - dual principal mismatch
  - delegated C2/C5 allowed cases
  - principal/jti 变更重放
- 验收：合法委托通过、非法复用全部拒绝

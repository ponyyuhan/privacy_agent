\section{Introduction}
\label{sec:intro}

Agentic systems increasingly execute real effects: sending messages, calling APIs, and installing third-party skills.
In this setting, runtime logic is exposed to prompt injection, malicious tool outputs, and supply-chain compromise.
The central systems question is no longer whether a policy exists, but whether effect execution is cryptographically bound to that policy decision.

A second tension appears in modern deployments.
To reduce leakage risk, operators insert centralized control planes (AI gateways, DLP filters, LLM firewalls), but these systems often inspect and retain sensitive request contents for auditing and operations \citep{azureapimllmlogs2026,cloudflarefirewallai2026,cloudflareaidlp2026,dynatracemasklogs2026}.
This creates a control-plane privacy paradox: the defense layer itself can become a privileged leakage sink.

SecureClaw addresses both problems with a single architecture: a non-bypassable execution boundary at commit time, plus an optional privacy-preserving outsourcing mode for policy checks.

\paragraph{Primary claim.}
For any external side effect, SecureClaw enforces that commit succeeds \emph{iff} the executor verifies dual policy proofs bound to the exact request tuple (including delegation context), freshness, and replay constraints.
In addition, plaintext confidentiality to the untrusted runtime is preserved by default handleization and explicit declassification.
In SAP mode, each single policy server view is simulatable from an explicit leakage function.

\paragraph{Contributions.}
\begin{itemize}
  \item \textbf{Non-bypassable effect boundary (NBE).} We provide a PREVIEW$\rightarrow$COMMIT protocol where an executor-side accept predicate is the sole effect gate, with dual-proof verification, request binding, and replay resistance.
  \item \textbf{End-to-end confidentiality and integrity composition.} We combine secret myopia (SM), patch-carrying egress integrity (PEI), skill-confined safety (SCS), and delegation-aware safety (DAS) into one compositional security statement with explicit assumptions.
  \item \textbf{Single-auditor privacy (SAP) for outsourced policy checks.} We compile policy evaluation into constant-shape PIR/MPC schedules and define an explicit leakage contract $L_{\mathsf{policy}}$ under which each single-server transcript is simulatable.
  \item \textbf{Claim-driven evidence.} We report official benchmark outcomes, transcript-distinguishability sweeps for SAP, and delegated multi-agent campaigns, aligned directly with theorem assumptions and residual risks.
\end{itemize}

\section{Problem Statement and Scope}
\label{sec:problem}

\subsection{Setting}

We consider agentic tool-use systems with five roles:
(1) an untrusted runtime $\mathcal{A}$,
(2) a trusted gateway $\mathcal{G}$,
(3) two policy servers $\mathcal{P}_0,\mathcal{P}_1$,
(4) a trusted executor $\mathcal{X}$,
and (5) a user $\mathcal{U}$ for explicit confirmations.
The adversary controls $\mathcal{A}$ and all skill code, and may submit arbitrary requests.

\subsection{Security objectives}

SecureClaw targets six objectives (five mandatory, one optional):
\begin{itemize}
  \item \textbf{NBE:} no side effect without valid dual commit proofs bound to the exact request.
  \item \textbf{SM:} sensitive plaintext is not exposed to $\mathcal{A}$ except explicit declassification.
  \item \textbf{PEI:} required sanitize patches are enforced before effect execution.
  \item \textbf{SCS:} skill bypasses are blocked under a verifiable capsule mediation contract.
  \item \textbf{DAS:} commit evidence is not replayable across caller/session/principal/delegation contexts.
  \item \textbf{SAP (optional):} each single policy server transcript is simulatable from $L_{\mathsf{policy}}$.
\end{itemize}

\subsection{Assumptions and non-goals}

\paragraph{Trusted base.}
Gateway and executor are trusted.
Compromise of either is out of scope.

\paragraph{Policy privacy assumption.}
SAP requires two policy servers and at least one non-colluding server.
If both collude, single-auditor privacy is impossible in this model.

\paragraph{System-level side channels.}
We do not claim protection against kernel microarchitectural channels.
Leakage claims are explicitly scoped to application-level channels and metadata captured by $L_{\mathsf{policy}}$ and $L_{\mathsf{sys}}$.

\paragraph{Design scope.}
We do not claim a universal MPC engine for arbitrary programs.
The artifact targets policy membership and fixed-circuit evaluation used by this threat model.

\section{Protocol and System Design}
\label{sec:protocol}

\subsection{Architecture}

SecureClaw separates planning from effects.
The runtime may propose actions, but only the executor can commit external side effects.
The gateway mediates policy preview and confidentiality controls; the executor enforces effect-time acceptance.

\subsection{Request binding and commit evidence}

For an effectful request, the gateway computes
\[
\mathsf{ReqHash}(\mathsf{intent\_id},\mathsf{caller},\mathsf{session},\mathsf{inputs},\mathsf{hctx})
:= \mathrm{SHA256}(\mathsf{Canon}(\cdot)),
\]
where
\[
\mathsf{hctx}:=(\mathsf{external\_principal},\mathsf{delegation\_jti}).
\]
$\mathsf{Canon}(\cdot)$ is deterministic canonical JSON serialization shared by gateway and executor.

PREVIEW returns a fresh action id and two commit-proof shares from distinct policy servers.
COMMIT succeeds only if the executor verifies both shares, recomputes the same request hash, checks freshness, enforces replay protection, and validates policy output semantics.

\subsection{Executor acceptance predicate}

Let $\mathsf{Accept}(\mathsf{req},\pi_0,\pi_1)$ denote executor acceptance.
Acceptance requires:
\begin{enumerate}
  \item dual proof parsing and MAC verification under distinct server identities;
  \item freshness inside configured TTL;
  \item context agreement on $(\mathsf{action\_id},\mathsf{program\_id},\mathsf{request\_sha256})$;
  \item recomputed request-hash equality including $\mathsf{hctx}$;
  \item replay-guard non-membership for action id;
  \item semantic bits satisfy allow/confirm/patch constraints.
\end{enumerate}
Any failure is fail-closed.

\subsection{Secret myopia and declassification}

Sensitive inputs are handleized at the gateway.
The untrusted runtime receives opaque handles and non-sensitive metadata, not plaintext.
Plaintext release requires explicit declassification, confirmation, and audit logging.
This realizes default confidentiality even when runtime logic is adversarial.

\subsection{Patch-carrying egress integrity}

If policy requires sanitization, patch constraints are cryptographically carried from preview outputs to commit semantics.
The executor applies the required sanitize transformation before side effects.
Thus, commit-time bypass by payload substitution is rejected by acceptance semantics.

\subsection{Skill ingress and capsule-mediated confinement}

Skill installation is treated as an effectful transaction under the same PREVIEW$\rightarrow$COMMIT boundary.
Runtime confinement assumptions are expressed by a capsule mediation contract (filesystem, network, process, transport constraints) with explicit downgrade semantics when assertions fail.

\subsection{Multi-agent federation and delegated authority}
\label{sec:multiagent}

Cross-agent delegation uses signed tuple
\[
\delta=(\mathsf{iss},\mathsf{sub},\mathsf{scope},\mathsf{session},\mathsf{exp},\mathsf{jti}).
\]
Gateway checks signature/session/caller/scope/revocation.
Effective authorization is capability intersection between internal caller and external principal.
Because $(\mathsf{external\_principal},\mathsf{delegation\_jti})$ is bound into $\mathsf{ReqHash}$, preview evidence cannot be replayed across principals or delegation contexts.

\subsection{Optional SAP mode: leakage-aware constant-shape outsourcing}

In SAP mode, policy checks are routed through fixed-shape PIR/MPC schedules.
Routing and schedule metadata are intentionally modeled by
$L_{\mathsf{policy}}=(L_{\mathsf{PIR}},L_{\mathsf{MPC}},L_{\mathsf{CONFIRM}},L_{\mathsf{TIME}})$.
Intent hiding is claimed only when public observables (e.g., program id, endpoint class, batch shape) are held constant across covered intents.

\section{Formal Security Claims and Proof Chain}
\label{sec:security}

Theorems and proofs are in Appendix \ref{app:proofs}; leakage formalization is in Appendix \ref{app:leakage}.

\paragraph{NBE theorem.}
No side effect is committed without dual valid request-bound commit proofs, up to MAC/hash/replay/time assumption failures.

\paragraph{SM theorem.}
Runtime plaintext advantage beyond explicit declassification is bounded by handle-guessing probability plus declared leakage.

\paragraph{PEI theorem.}
Accepted commits cannot bypass required sanitize semantics unless implementation diverges from the specified accept predicate.

\paragraph{SCS theorem.}
Under capsule contract assumptions and NBE+SM, forbidden direct bypass actions are blocked.

\paragraph{DAS theorem.}
Cross-context reuse of preview evidence is rejected unless cryptographic binding or delegation-validation assumptions fail.

\paragraph{SAP theorem (optional).}
Under PIR single-server security, HBC 2PC privacy, non-collusion, and fixed-shape deployment, each single policy server view is simulatable from $L_{\mathsf{policy}}$.

\paragraph{Compositional statement.}
The overall system bound is a union bound over NBE/SM/PEI/SCS/DAS/(optional SAP) bad events with assumption-partitioned dependencies.

\section{Evaluation Methodology}
\label{sec:eval}

\subsection{Research questions}

\begin{itemize}
  \item \textbf{RQ1 (Effect integrity):} Does SecureClaw eliminate attack-side leakage while preserving benign utility on official benchmarks?
  \item \textbf{RQ2 (Policy privacy):} Does transcript distinguishability collapse under full SAP shaping?
  \item \textbf{RQ3 (Federation safety):} Are delegated multi-agent workflows enforced without cross-context replay?
  \item \textbf{RQ4 (Systems cost):} What latency overhead is introduced by enforcement and shaping?
\end{itemize}

\subsection{Benchmarks and baselines}

Official benchmark coverage is C1--C5.
Channels C6--C7 are evaluated separately via synthetic suites and are not mixed into official-rate tables.

We report:
(1) SecureClaw variants (full, policy-only, sandbox-only, single-server-policy),
(2) native runtimes (Codex native, OpenClaw native), and
(3) mediation defenses on native outputs (DRIFT, IPIGuard, AgentArmor) \citep{drift2025,ipiguard2025,agentarmor2025}.

Native baselines use a neutral task prompt (no explicit anti-leak instructions) to isolate platform behavior rather than prompt-tuning effects.

\subsection{Metrics}

For attack set $\mathcal{D}_{\mathsf{atk}}$:
\[
\mathsf{attack\_leak\_rate}(S)=\frac{|\{d\in\mathcal{D}_{\mathsf{atk}}: \mathsf{leaked}(d,S)=1\}|}{|\mathcal{D}_{\mathsf{atk}}|}.
\]
We report attack block rate, benign allow rate, Wilson intervals, and two-sided Fisher exact tests versus SecureClaw full.

For SAP, we run transcript distinguishability sweeps using mutual information (MI) and classifier MAP accuracy over single-server metadata views.

SecureClaw full reaches zero observed attack leakage with full attack blocking on the official set.

with all three at $100\%$ benign allow in this run.
These methods improve mediated runtime behavior but do not provide an executor-side non-bypassable commit boundary.

\subsection{RQ3: Delegated multi-agent safety}
\label{sec:multiagent-eval}

A deterministic delegated workflow campaign (7 cases) reports 7/7 expected outcomes, including missing-delegation denial, explicit revocation, revoked-token denial, dual-principal denial, and delegated memory mediation.
Gateway-level check latency is low (median $0.41$ ms, p95 $6.28$ ms).

\section{Performance and Deployability}
\label{sec:perf}

\subsection{Cost decomposition}

Constant-shape privacy introduces three overhead sources:
(1) dummy compute from padding/cover traffic,
(2) mixing queueing delay,
and (3) transport framing overhead.
The executor-side enforcement path remains constant-sized and is dominated by dual MAC verification and replay-guard lookup.

\subsection{Deployment requirements}

Production deployment requires:
(1) persistent replay state for restart-safe replay protection,
(2) key rotation with explicit key identifiers,
(3) tamper-evident audit chaining,
and (4) consistent canonical binding shared by gateway and executor.

\section{Limitations and Failure Modes}
\label{sec:limits}

SecureClaw does not claim security if the trusted gateway or executor is compromised.
SAP does not hold under full policy-server collusion.
SCS is conditional on capsule-contract validity.
Kernel-level side channels and broad availability failures are outside the formal security claim and are treated as operational risks.

\section{Related Work and Positioning}
\label{sec:related}

Prompt-injection and tool-chain attacks in LLM agents are now well documented \citep{greshake2023indirect,injecagent2024,toolselect2025,stac2025}.
Benchmark suites quantify leakage and robustness across channels and environments \citep{agentdojo2024,asb2024,agentleak2026,magpie2025,topbench2025,wasp2025,vpibench2025,agentharm2024}.
Defense lines such as DRIFT, IPIGuard, and AgentArmor focus on runtime mediation and detection \citep{drift2025,ipiguard2025,agentarmor2025}.

Faramesh formalizes an Action Authorization Boundary (AAB) as a non-bypassable execution concept \citep{faramesh2026}.
SecureClaw is complementary but distinct in two ways:
(1) it explicitly composes AAB-style effect integrity with runtime plaintext confinement and delegation-context hash binding (SM/PEI/DAS), and
(2) it adds optional single-auditor privacy for outsourced policy evaluation via an explicit leakage contract and transcript-simulation claim (SAP), rather than exposing policy queries to a single control plane.

At the protocol layer, MCP/A2A/AP2 and OAuth hardening define interoperability and delegated authorization interfaces \citep{mcpauth2025,mcpsecbp2025,a2aspec2025,a2aproto2025,ap2spec2025,rfc6819,rfc9700,rfc9449,rfc8705}.
SecureClaw focuses on a different guarantee: execution-time cryptographic binding between authorization and committed effects under adversarial runtime control.
